다수의 장비가 동시에 접속하는 실시간 서버에서 서버·네트워크 쪽을 주로 다루고, 새 기능 개발부터 운영 배포까지 합니다.

백엔드에서 서버와 네트워크 통신을 주로 맡고 있습니다. 여러 장비가 한꺼번에 붙는 환경이라, 한 곳이 막혀도 전체가 멈추지 않게 만드는 게 중요합니다. 운영 중 생기는 연결 관리나 동시성 문제는 로그와 코드로 원인을 찾아 고치고, 새 기능은 설계부터 배포까지 직접 만듭니다.

한 일

실시간 서버 리팩토링과 네트워크 통신 안정화

Java 17 · Netty · 실시간 통신 · 안정화

다수의 장비가 붙는 실시간 서버의 통신과 구조를 계속 손봤습니다. 재직하면서 가장 오래 붙잡고 있는, 제가 주도한 작업입니다.

얽힌 코드와 중복 로직을 정리했습니다. 길어진 메서드는 책임별로 나누고, 사용하지 않는 코드와 중복을 제거했습니다. 초기화 흐름도 하나로 정리했습니다. 외부 장비로 나가는 XML을 문자열로 조립하던 코드는 타입 모델로 바꿔 오타·누락 위험을 줄였고, 외부 서버와의 통신은 스레드를 오래 붙잡는 동기 방식이라 비동기로 바꿔 처리량을 늘렸습니다.

응답하지 않는 외부 장비 하나가 공용 스레드풀을 장시간 점유하면서, 정상 요청까지 느려지거나 실패하던 문제가 있었습니다. 무거운 로직·DB 작업을 통신 처리(이벤트 루프)에서 떼어 전용 실행 그룹으로 돌리고, 작업 종류별로 풀을 격리했으며, 처리 큐에 상한을 둬 넘치면 거부 로그를 남기게 했습니다. 문제 장비로 가는 호출은 서킷브레이커로 자동 제한했습니다.

특정 이벤트 요청이 몰릴 때 서버 응답이 급락하던 문제도 내부 HTTP 호출을 비동기로 바꾸고 타임아웃을 적용해 개선했습니다. 부하 시험에서는 병목이 있던 흐름을 큰 폭으로 끌어올렸습니다.

연결 누적 추적 사례 — 장비가 비정상 종료될 때 소켓 연결이 정상적으로 정리되지 않아 종료 대기 상태가 누적되고, 스레드 수·메모리가 함께 늘던 문제가 있었습니다. 네트워크·프로세스 상태와 스레드 상태를 함께 봐서 원인을 좁혔습니다. 채널 생명주기 기준으로 종료 처리를 보강하고 유휴 연결은 자동 정리, 요청마다 ID를 붙여 전 구간을 로그로 추적할 수 있게 했습니다.

선택과 이유
서킷브레이커는 라이브러리 대신 직접 구현 — 외부 의존성을 더하지 않고, 장비 식별 단위마다 가볍게 붙는 형태로 기존 스레드풀 구조에 맞췄습니다.
풀은 서버별이 아니라 작업 종류별로 격리 — 연동 서버 수만큼 풀을 두면 관리가 안 되므로, 종류별로 나눠 서버가 늘어도 구조가 그대로이게 했습니다.
XML은 문자열 조립 대신 타입 모델 — String.format 조립은 오타·누락이 실행 중에야 터져서, 타입으로 바꿔 만드는 단계에서 걸리게 했습니다.
측정: 서버에 요청을 한꺼번에 보내 응답 시간과 성공 여부를 확인 · 빠른 요청과 외부 지연이 섞인 느린 요청을 함께 검증 · 제한된 리소스 환경에서 재현
응답 지연 장비가 섞인 부하 · 일반 요청 평균 응답큰 폭 개선
before 초 단위 지연
after 밀리초 단위 응답
제한된 리소스 상황 · 실패하던 요청을 정상 처리정상화
실패

외부 이벤트 감지·알림 기능 개발 (0→1)

Spring Boot · REST · 외부 관제 서버 연동

외부 관제 서버가 보내는 이벤트를 받아 판정하고, 여러 채널로 알림을 보내는 기능을 새로 만들었습니다. API 설계부터 배포까지 백엔드는 제가 맡았고, 모바일앱·사내 백오피스·외부 연동사와 인터페이스를 맞췄습니다.

설정·문구 관리, 이벤트 조회, 수동·자동 알림, 관리자 대시보드 API를 만들고 관제 서버와 양방향으로 연동했습니다. 이벤트가 발생하면 모바일앱과 장비 단말로 알림이 전달되도록 했습니다.

연동 서버·모바일앱·관리자 화면이 필요로 하는 API가 서로 달라서, 소비자별로 엔드포인트를 나눠 한쪽을 고쳐도 다른 쪽에 영향이 가지 않게 했습니다.

이벤트 처리외부 서버 연동알림관리자 대시보드

대용량 조회 성능 최적화 (DB)

MariaDB · MySQL · 파티셔닝 · 인덱스 · 쿼리 튜닝

데이터가 크게 누적되면서 조회 지연이 발생했습니다. 실행 계획(EXPLAIN)으로 느린 쿼리를 짚고, 누적 데이터 테이블은 파티셔닝 + 인덱스 재설계, 이력 조회 테이블은 페이징 조회로 바꿨습니다.

‘쿼리는 빠른데 API는 느린’ 경우도 있어서, SQL 실행 시간만 보지 않고 커넥션 풀(HikariCP) 대기·DB 응답 전송 지연까지 나눠서 병목이 어디인지 확인했습니다.

주요 조회 API는 초 단위 지연에서 밀리초 단위 응답으로 개선했고, 불필요한 응답 필드를 줄여 전송 데이터 크기도 크게 낮췄습니다.

→ 운영 환경과 유사한 누적 데이터 기준으로 측정한 결과입니다.

파일 업로드 중계 게이트웨이

Spring Boot · HTTP multipart · 외부 처리 서버 연동 · Redis

모바일·웹에서 올린 파일을 외부 처리 서버와 장비 단말로 중계하는 게이트웨이를 만들었습니다. 큰 파일을 안정적으로 주고받는 게 관건이었습니다.

HTTP multipart로 주고받고, 단말 저장 공간이 부족하면 잘라서 나눠 보냅니다. 하루 업로드 용량·개수·크기 제한은 Redis로 관리하고, 전송 이력도 남깁니다.

외부 장비 간 실시간 연동

Java · Spring Boot · 실시간 이벤트

서로 다른 외부 장비가 상태를 주고받고 특정 동작을 요청할 수 있도록, 장비 사이를 잇는 연동을 개발했습니다. 호출 값과 이동 상태를 처리하고, 예상치 못한 값이 들어와도 서버가 중단되지 않도록 검증 로직을 보강했습니다.

규격 검증과 코드 정상화

Java · 계약 테스트 · 규격 대조

기존 구현체를 Java 기반으로 전환하는 과정에서, 저는 해당 코드를 기기 연동 프로토콜 문서 기준으로 테스트하고 개선했습니다.

XML 기반 프로토콜 코드를 골든 케이스로 검증하는 테스트를 만들어 빌드·파싱·응답 단계를 확인하고, 모바일 API 여러 도메인의 동작을 표로 정리했습니다. 예시와 정의가 일치하지 않는 부분은 문서의 정의를 기준으로 검증하도록 했습니다. 문서와 구현을 자동으로 대조하는 검증기도 프로토타입으로 만들었고, 상태 표시 오작동도 원인을 찾아 개선했습니다.

운영 · 배포 · 안정성 개선

운영 배포·검증
IoT 서버 신규 버전을 여러 운영 환경에 적용했습니다. ‘배포 성공’과 ‘서비스 정상 동작’은 다르다는 걸 겪고, 버전 확인·연결 상태·주요 연동 API 점검까지 포함한 배포 검증 절차를 만들었습니다. 배포 자동화 도구를 활용해 대상 환경별 선택 배포와 실패 이력 확인 절차도 정리했습니다.
장비 상태 동기화 개선
출입 관련 장비와 서버 간 상태 불일치로 발생한 동기화 문제를 분석하고, 재동기화 절차를 보강해 운영 안정성을 개선했습니다. 서버만이 아니라 장비 담당 조직과 함께 원인을 나누어 확인했습니다.
